# 03. Row vs Column Oriented Databases 학습 노트

## 개요

이번 노트는 부하 테스트 결과 데이터를 저장하고 집계하는 방식과 연결해서 Row-oriented database와 Column-oriented database의 차이를 정리한다.

핵심은 데이터베이스가 디스크에서 데이터를 읽는 최소 단위인 block에 어떤 형태로 데이터를 모아 두느냐에 있다.

```text
Row-oriented    -> 한 행의 여러 컬럼 값을 함께 저장
Column-oriented -> 같은 컬럼의 값들을 함께 저장
```

따라서 어떤 방식이 더 좋은지는 절대적인 문제가 아니라, 애플리케이션이 어떤 쿼리를 자주 수행하는지에 따라 달라진다.

## 예시 테이블

부하 테스트 요청 로그를 단순화하면 아래와 같은 형태로 볼 수 있다.

```text
request_index | virtual_user_id | latency_ms | status_code | success | error_message
1             | 1               | 120        | 200         | true    | null
2             | 1               | 180        | 500         | false   | SIMULATED_5XX
3             | 2               | 95         | 200         | true    | null
```

우리 과제에서는 이 요청 단위 로그를 Parquet 파일로 저장한 뒤 DuckDB로 집계한다.

```sql
SELECT
  count(*) AS total_requests,
  count(*) FILTER (WHERE success) AS success_count,
  avg(latency_ms) AS average_latency_ms,
  quantile_cont(latency_ms, 0.95) AS p95_latency_ms,
  max(latency_ms) AS max_latency_ms
FROM read_parquet(...)
```

이 쿼리는 전체 행을 자세히 보는 것이 아니라 `success`, `latency_ms` 같은 특정 컬럼을 많이 읽고 집계한다. 이 지점에서 column-oriented 저장 방식이 왜 유리한지 이해할 수 있다.

## Row-oriented Database

Row-oriented database는 한 행에 속한 여러 컬럼 값을 가까운 위치에 저장한다.

예를 들어 요청 로그가 아래처럼 행 단위 block에 저장된다고 생각할 수 있다.

```text
block 1
1 | 1 | 120 | 200 | true  | null
2 | 1 | 180 | 500 | false | SIMULATED_5XX

block 2
3 | 2 | 95  | 200 | true  | null
4 | 2 | 140 | 200 | true  | null
```

### 장점

특정 요청 1건의 전체 정보를 조회할 때 효율적이다.

```sql
SELECT *
FROM request_logs
WHERE request_index = 2;
```

한 행을 찾으면 그 행의 `virtual_user_id`, `latency_ms`, `status_code`, `success`, `error_message`가 같이 붙어 있으므로 한 번에 읽기 좋다.

또한 일반적인 트랜잭션 처리에도 잘 맞는다.

- 사용자 1명 조회
- 주문 1건 조회
- 테스트 실행 1건 상세 조회
- 상태 변경 후 저장

우리 과제의 `test_runs`, `test_run_results` 같은 테이블은 이런 성격에 가깝다. 실행 목록, 실행 상세, 상태 변경처럼 행 단위로 읽고 쓰는 일이 중심이기 때문이다.

### 단점

집계 쿼리에는 불필요한 데이터까지 함께 읽는 비용이 생긴다.

```sql
SELECT avg(latency_ms)
FROM request_logs;
```

이 쿼리는 사실 `latency_ms`만 필요하다. 하지만 row-oriented 저장 방식에서는 같은 행에 들어 있는 `virtual_user_id`, `status_code`, `success`, `error_message` 같은 값도 같은 block에 있으므로 같이 읽히기 쉽다.

데이터가 작을 때는 큰 차이가 없지만, 요청 로그가 수십만 건 또는 수백만 건으로 늘어나면 불필요한 I/O가 커진다.

## Column-oriented Database

Column-oriented database는 같은 컬럼의 값들을 가까운 위치에 저장한다.

요청 로그를 컬럼 단위로 보면 아래처럼 생각할 수 있다.

```text
request_index
1 | 2 | 3 | 4

latency_ms
120 | 180 | 95 | 140

success
true | false | true | true

status_code
200 | 500 | 200 | 200
```

### 장점

특정 컬럼만 읽는 분석/집계 쿼리에 효율적이다.

```sql
SELECT
  avg(latency_ms),
  max(latency_ms)
FROM request_logs;
```

이 경우 `latency_ms` 컬럼만 읽으면 된다. 실패 수를 계산할 때도 `success` 컬럼 중심으로 읽으면 된다.

우리 과제의 Parquet 저장과 DuckDB 집계가 이 방향에 가깝다.

- raw request log는 Parquet 파일로 저장한다.
- Parquet은 컬럼 기반 파일 포맷이다.
- DuckDB는 `read_parquet()`으로 필요한 컬럼을 읽어 집계할 수 있다.
- 평균, p95, 최대 latency 같은 통계 계산에 잘 맞는다.

즉, 요청 단위 로그 전체를 매번 DB row로 읽는 대신, 분석에 필요한 컬럼 중심으로 읽어 요약 결과를 계산하는 구조다.

### 대량 쓰기에 유리한 이유

Column-oriented 방식은 단건 insert에는 불리할 수 있지만, append 중심의 대량 쓰기에는 유리해질 수 있다.

핵심은 데이터를 한 건씩 즉시 저장하는 것이 아니라, 일정량을 모아서 컬럼 단위로 한 번에 기록한다는 점이다.

예를 들어 요청 로그 10만 건을 저장한다고 생각해 보자.

```text
row-oriented
요청 1번 전체 컬럼 저장
요청 2번 전체 컬럼 저장
요청 3번 전체 컬럼 저장
...

column-oriented
latency_ms 컬럼 값 10만 개 저장
success 컬럼 값 10만 개 저장
status_code 컬럼 값 10만 개 저장
...
```

같은 컬럼의 값들은 데이터 타입과 패턴이 비슷하다. 이 특성 때문에 대량 적재 시 다음 이점이 생긴다.

- 같은 타입의 값을 연속으로 쓰기 때문에 압축 효율이 좋다.
- `success`, `status_code`처럼 반복되는 값은 더 작게 저장할 수 있다.
- 컬럼별로 큰 덩어리를 순차적으로 기록할 수 있어 디스크 쓰기 패턴이 단순해진다.
- 분석에 필요 없는 컬럼까지 매번 읽지 않도록, 저장 단계부터 컬럼 단위 구조를 만들 수 있다.

우리 과제의 요청 로그도 비슷하다. worker가 요청 결과를 한 건씩 API 응답용 DB row로 계속 저장하는 대신, 실행이 끝난 뒤 생성된 request log 배열을 Parquet 파일로 한 번에 저장한다.

```text
request log 여러 건 생성
-> DuckDB in-memory table 적재
-> Parquet 파일로 bulk write
-> DuckDB read_parquet()으로 집계
```

이 구조에서는 개별 요청 1건을 즉시 조회하는 성능보다, 대량 로그를 효율적으로 저장하고 나중에 집계하는 성능이 더 중요하다. 그래서 column-oriented 파일 포맷인 Parquet이 잘 맞는다.

### 단점

한 행의 모든 컬럼을 자주 조회하는 경우에는 오히려 비효율적일 수 있다.

```sql
SELECT *
FROM request_logs
WHERE request_index = 2;
```

이 쿼리는 한 요청의 모든 컬럼이 필요하다. Column-oriented 방식에서는 여러 컬럼 묶음에서 같은 행 위치의 값을 다시 조합해야 하므로 row-oriented 방식보다 부담이 커질 수 있다.

쓰기 관점에서도 단순하지 않다. 행 하나를 추가할 때 여러 컬럼 저장 구조에 값을 나눠 넣어야 하기 때문이다.

따라서 "column-oriented는 쓰기가 항상 빠르다"라고 이해하면 안 된다. 단건 row insert가 많은 OLTP 작업에는 부담이 될 수 있고, append-only 또는 batch write 중심의 분석 로그 적재에서 강점이 커진다고 보는 것이 정확하다.

## 우리 과제에 적용해 보기

현재 과제는 두 종류의 데이터를 다룬다.

### 실행 상태 데이터

```text
test_runs
test_run_results
```

이 데이터는 row-oriented RDB 테이블에 잘 맞는다.

- 실행 1건 단위로 생성된다.
- 상태 변경이 자주 일어난다.
- API에서 실행 상세를 행 단위로 조회한다.
- 결과 요약도 실행 1건에 매핑된다.

따라서 일반적인 RDB 테이블로 관리하는 것이 자연스럽다.

### 요청 단위 로그 데이터

```text
data/test-runs/{testRunId}/requests.parquet
```

이 데이터는 column-oriented 파일 포맷에 잘 맞는다.

- 요청 로그 row 수가 많아질 수 있다.
- 주된 관심사는 개별 요청 상세보다 통계 집계다.
- `latency_ms`, `success`, `error_message`처럼 일부 컬럼만 읽어도 결과를 만들 수 있다.
- 평균, p95, 최대값, 실패 요약 계산이 중요하다.

그래서 우리 과제에서 Parquet과 DuckDB를 사용하는 선택은 "raw log는 분석용 컬럼 기반 저장소에 가깝게 두고, API 응답용 요약 결과는 RDB에 저장한다"는 구조로 이해할 수 있다.

## 정리

Row-oriented database는 행 전체를 자주 읽고 쓰는 트랜잭션성 작업에 적합하다.

Column-oriented database는 많은 데이터에서 일부 컬럼만 읽어 집계하는 분석성 작업에 적합하다.

우리 과제 기준으로는 아래처럼 나눠 생각할 수 있다.

```text
실행 요청/상태/요약 결과 -> RDB 테이블
요청 단위 대량 로그       -> Parquet
통계 집계                 -> DuckDB + read_parquet()
```

결국 중요한 기준은 저장 기술의 이름이 아니라 쿼리 패턴이다.

- `SELECT *`로 특정 실행 1건을 자주 본다면 row-oriented가 유리하다.
- `avg(latency_ms)`, `p95_latency_ms`, 실패 수처럼 특정 컬럼을 대량 집계한다면 column-oriented가 유리하다.
