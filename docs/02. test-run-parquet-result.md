# 02. TestRun Parquet 데이터 적재 및 결과 집계 정리

## 개요

이번 작업은 `TestRunsWorker`의 시뮬레이션 실행 위치에 실제 데이터 처리 흐름을 추가한 변경이다.

아직 실제 HTTP 부하 테스트를 수행하지는 않고, 결정적인 규칙으로 생성한 요청 로그를 Parquet 파일로 저장한 뒤 DuckDB SQL로 집계해 `TestRunResult`를 저장한다.

처리 범위는 아래와 같다.

```text
PENDING -> RUNNING -> SUCCESS
                 \-> FAILED
```

성공 시에는 요청 단위 로그가 Parquet 파일로 남고, 요약 결과는 `test_run_results` 테이블에 저장된다.

## 처리 흐름

1. `TestRunsWorker`가 가장 오래된 `PENDING` 작업 1건을 조회한다.
2. 작업은 Entity 메서드로 `RUNNING` 상태가 되고 `startedAt`이 기록된다.
3. `runSimulation(testRun)`이 요청 단위 시뮬레이션 데이터를 생성한다.
4. `TestRunParquetService`가 요청 로그를 DuckDB in-memory 테이블에 적재한다.
5. DuckDB `COPY ... TO ... (FORMAT parquet)`로 요청 로그를 Parquet 파일에 저장한다.
6. DuckDB `read_parquet()` SQL로 결과 요약을 계산한다.
7. `TestRunsRepository.saveResult()`가 `TestRunResult`를 저장한다.
8. 작업은 Entity 메서드로 `SUCCESS` 상태가 되고 `finishedAt`이 기록된다.
9. 처리 중 예외가 발생하면 Entity 메서드로 `FAILED` 상태가 되고 `errorMessage`, `finishedAt`이 기록된다.

## 주요 구성

### TestRunsWorker

`TestRunsWorker`는 기존 상태 전이 흐름에 시뮬레이션, Parquet 저장, 결과 저장 책임을 연결한다.

핵심 책임은 다음과 같다.

- 중복 tick 방지를 위한 in-memory lock 관리
- `PENDING` 작업 조회 및 `RUNNING` 상태 저장
- 요청 로그 시뮬레이션 데이터 생성
- Parquet 저장 및 집계 서비스 호출
- `TestRunResult` 저장
- 성공 시 `SUCCESS`, 실패 시 `FAILED` 상태 저장

시뮬레이션 데이터는 네트워크 호출 없이 생성한다.

- 총 요청 수: `virtualUsers * durationSec`
- latency: 입력값과 index 기반의 deterministic formula 사용
- 실패 샘플: 25번째 요청마다 `SIMULATED_5XX` 실패로 처리

### TestRunParquetService

`TestRunParquetService`는 Parquet 파일 저장과 DuckDB 집계를 담당한다.

DuckDB는 영구 테이블을 만들지 않고 실행마다 in-memory instance를 사용한다.

```ts
duckdb.DuckDBInstance.create(':memory:');
```

따라서 test id별로 DuckDB 테이블이 남는 구조가 아니라, test id별 Parquet 파일만 남는다.

```text
data/test-runs/{testRunId}/requests.parquet
```

`TEST_RUN_PARQUET_DIR` 환경변수가 있으면 해당 경로를 Parquet 루트 디렉터리로 사용한다.

```text
TEST_RUN_PARQUET_DIR=/tmp/load-test-results
-> /tmp/load-test-results/{testRunId}/requests.parquet
```

### Parquet 요청 로그 스키마

Parquet 파일에는 요청 단위 로그를 저장한다.

```text
test_run_id
request_index
virtual_user_id
started_at
finished_at
latency_ms
status_code
success
error_message
```

집계 결과는 Parquet 파일을 다시 읽어서 계산한다.

```sql
SELECT
  count(*) AS total_requests,
  count(*) FILTER (WHERE success) AS success_count,
  count(*) FILTER (WHERE NOT success) AS failure_count,
  avg(latency_ms) AS average_latency_ms,
  quantile_cont(latency_ms, 0.95) AS p95_latency_ms,
  max(latency_ms) AS max_latency_ms
FROM read_parquet(...)
```

DuckDB의 `count(*)` 결과는 JavaScript에서 `bigint`로 반환될 수 있으므로 저장 입력값을 만들 때 `Number(...)`로 변환한다.

### TestRunResult 저장

`TestRunsRepository.saveResult()`는 집계 결과를 `test_run_results` 테이블에 저장한다.

저장되는 주요 값은 다음과 같다.

- `totalRequests`
- `successCount`
- `failureCount`
- `averageLatencyMs`
- `p95LatencyMs`
- `maxLatencyMs`
- `failureSummaryJson`

실패 요약은 `{ reason, count }[]` 형태를 JSON 문자열로 저장한다.

API 응답 시에는 `failureSummaryJson`을 다시 파싱하고, latency 계열 값은 소수점 한 자리로 반올림한다.

### TestRun Entity

실패 상태 전이도 Entity 메서드로 관리한다.

```ts
testRun.fail(errorMessage, finishedAt);
```

허용 규칙은 다음과 같다.

- `start()`: `PENDING` 상태에서만 `RUNNING`으로 변경
- `complete()`: `RUNNING` 상태에서만 `SUCCESS`로 변경
- `fail()`: `RUNNING` 상태에서만 `FAILED`로 변경

잘못된 상태 전이는 일반 `Error`를 던진다.

## 현재 의도적으로 제외한 것

이번 작업에서는 아래 항목을 구현하지 않았다.

- 실제 HTTP 요청 기반 부하 테스트
- Parquet raw log 조회 API
- Parquet 파일 경로를 `TestRunResult`에 저장하는 컬럼
- 실패 작업 재시도 정책
- 다중 서버/다중 워커 환경의 DB row lock 처리

## 테스트 범위

추가 및 변경된 테스트는 크게 세 축이다.

### Entity 테스트

- `RUNNING` 작업이 `fail()` 호출 후 `FAILED`가 되는지 확인
- 실패 처리 시 `errorMessage`, `finishedAt`이 기록되는지 확인
- 잘못된 상태에서 `fail()`을 호출하면 에러가 나는지 확인

### Worker 테스트

- 작업이 있으면 `RUNNING` 저장 후 Parquet 집계 결과를 저장하고 `SUCCESS` 상태를 저장하는지 확인
- Parquet 저장 또는 집계 중 에러가 나면 `FAILED` 상태로 저장하는지 확인
- 처리 중 tick이 다시 들어와도 중복 조회/처리를 하지 않는지 확인

### Parquet Service 테스트

- 요청 로그가 Parquet 파일로 저장되는지 확인
- DuckDB `read_parquet()`으로 저장된 row count를 확인
- `totalRequests`, `successCount`, `failureCount`, latency 집계, 실패 요약이 계산되는지 확인
