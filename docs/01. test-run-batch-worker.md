# 01. TestRun 배치 상태 전이 정리

## 개요

이번 작업은 부하 테스트 실행 요청이 생성된 뒤, 별도 worker가 `PENDING` 상태의 작업을 비동기적으로 처리하도록 만든 변경이다.

현재 범위는 실제 부하 테스트 시뮬레이션과 결과 저장이 아니라, 실행 상태를 아래 순서로 안전하게 전환하는 데 집중한다.

```text
PENDING -> RUNNING -> SUCCESS
```

## 처리 흐름

1. 사용자가 `POST /test-runs`로 테스트 실행을 요청한다.
2. API는 `test_runs` 테이블에 `PENDING` 상태로 실행 요청을 저장하고 즉시 응답한다.
3. `TestRunsWorker`가 1초마다 가장 오래된 `PENDING` 작업 1건을 조회한다.
4. 조회된 작업은 Entity 메서드로 `RUNNING` 상태가 되고 `startedAt`이 기록된다.
5. 시뮬레이션 자리만 남겨둔 `runSimulation()`이 실행된다.
6. 작업은 Entity 메서드로 `SUCCESS` 상태가 되고 `finishedAt`이 기록된다.

## 주요 구성

### ScheduleModule

`@nestjs/schedule`을 사용해 주기 실행을 등록했다.

```ts
ScheduleModule.forRoot();
```

`TestRunsModule`에서 schedule 기능을 활성화하고, `TestRunsWorker`를 provider로 등록한다.

### TestRunsWorker

`TestRunsWorker`는 1초마다 `PENDING` 작업을 처리한다.

핵심 책임은 다음과 같다.

- 중복 tick 방지를 위한 in-memory lock 관리
- 가장 오래된 `PENDING` 작업 조회
- `RUNNING` 상태 저장
- 시뮬레이션 실행 위치 제공
- `SUCCESS` 상태 저장

중복 실행 방지는 단일 프로세스 기준으로 `isProcessing` 플래그를 사용한다.

```ts
if (this.isProcessing) {
  return;
}
```

현재 프로젝트 가정이 단일 서버/단일 워커이므로 DB row lock은 적용하지 않았다.

### TestRun Entity

상태 변경 규칙은 서비스나 worker가 직접 필드를 바꾸지 않고 Entity 내부 메서드로 관리한다.

```ts
testRun.start(startedAt);
testRun.complete(finishedAt);
```

각 메서드는 허용된 상태에서만 실행된다.

- `start()`: `PENDING` 상태에서만 `RUNNING`으로 변경
- `complete()`: `RUNNING` 상태에서만 `SUCCESS`로 변경

잘못된 상태 전이는 일반 `Error`를 던진다. Entity가 NestJS 프레임워크 예외에 의존하지 않도록 하기 위한 선택이다.

### TestRunsRepository

배치 처리를 위해 다음 메서드를 추가했다.

- `findOldestPending()`: 가장 오래된 `PENDING` 작업 1건 조회
- `save(testRun)`: Entity 상태 변경 결과 저장

조회 정렬은 먼저 요청 시각, 그 다음 id 기준이다.

```text
requestedAt ASC, id ASC
```

## 현재 의도적으로 제외한 것

이번 작업에서는 아래 항목을 구현하지 않았다.

- 실제 부하 테스트 시뮬레이션 로직
- `TestRunResult` 저장
- 실패 시 `FAILED` 상태 전환
- 재시도 정책
- 다중 서버/다중 워커 환경의 DB lock 처리

## 테스트 범위

추가된 테스트는 크게 두 축이다.

### Entity 테스트

- `PENDING` 작업이 `start()` 호출 후 `RUNNING`이 되는지 확인
- `RUNNING` 작업이 `complete()` 호출 후 `SUCCESS`가 되는지 확인
- 잘못된 상태에서 상태 전이를 시도하면 에러가 나는지 확인

### Worker 테스트

- `PENDING` 작업이 없으면 저장하지 않는지 확인
- 작업이 있으면 `RUNNING` 저장 후 `SUCCESS` 저장 순서가 지켜지는지 확인
- 처리 중 tick이 다시 들어와도 중복 조회/처리를 하지 않는지 확인

## 이후 확장 방향

다음 단계에서는 `runSimulation()` 내부에 실제 실행 로직을 넣고, 완료 시 `TestRunResult`를 저장하면 된다.

실패 처리를 추가할 때는 `TestRun` Entity에 `fail(errorMessage, finishedAt)` 같은 상태 전이 메서드를 추가해 `FAILED` 상태 변경도 같은 방식으로 관리하는 것이 좋다.
